## 🔴 CRÍTICO #1: `next/og` SÍ funciona con static export

**Evidencia:** `app/apple-icon.tsx` usa esto:

```tsx
import { ImageResponse } from "next/og";
export const dynamic = "force-static";
```

Y el build estático produce `/public/apple-icon.png` sin runtime. **Por tanto** las OG images dinámicas por artículo se pueden implementar con archivos `opengraph-image.tsx` por ruta — sin necesidad de runtime ni de librerías externas tipo `satori`+`resvg`.

Esto invalida la nota de auto-revisión del Plan C que argumenta "next/og requiere runtime serverless". **Plan E propuesto a continuación** corrige esto.

## 🔴 CRÍTICO #2 + #3: ToC (Plan B3) requiere rehype-slug

**Estado actual** (`components/mdx.tsx`):

```tsx
<ReactMarkdown remarkPlugins={[remarkGfm]} ...>
```

`remarkGfm` **no añade IDs a los headings**. Sin `rehype-slug`, los `<h2>Sección</h2>` se renderizan sin `id`, y los enlaces del ToC (`#seccion`) no apuntan a nada.

**Fixes necesarios:**

1. **Añadir dep:** `npm install rehype-slug`
2. **Pipeline MDX:** `<ReactMarkdown rehypePlugins={[rehypeSlug]}>` (con `rehype-slug` ya aplicado en build, los IDs se generan server-side)
3. **Algoritmo de slugify** debe **coincidir exactamente** con el de `rehype-slug`:
   - `rehype-slug` v6+ usa: `text.toLowerCase().trim().replace(/[\s\W-]+/g, '-')`
   - Esto **preserva acentos** (`¿` queda como parte del slug si va entre chars válidos). Probablemente queramos normalizar a ASCII para SEO con audiencias hispanohablantes — pero eso requiere reemplazar `rehype-slug` por un plugin custom o por ambos.
   - **Decisión recomendada:** usar el algoritmo por defecto de `rehype-slug` (preserva acentos) y añadir `rehype-slug` con la opción `prefix: ''` por defecto. Si consistencia con SEO es prioridad, en una Task futura se puede usar `rehype-slug` con `slugger` + función custom.

## 🟠 GRAVE #6: SHA256 del CSS — método robusto

**Problema:** `grep -oP "(?<=__html: \`).*(?=\`)"`asume single-line. Si la CSS se multi-linea o si varios`<style>` similar se añaden, falla.

**Fix:** Script Node.js determinístico que extrae la string UTF-8 exacta:

```ts
// scripts/hash-inline-style.ts
import fs from "node:fs";
import crypto from "node:crypto";

const layout = fs.readFileSync("app/layout.tsx", "utf-8");
const matches = [...layout.matchAll(/dangerouslySetInnerHTML=\{\{ __html: `([^`]+)`/g)];
if (matches.length === 0) throw new Error("No inline style/script blocks found");

for (let i = 0; i < matches.length; i++) {
  const content = matches[i][1];
  const hash = crypto.createHash("sha256").update(content).digest("base64");
  console.log(`BLOCK[${i}] sha256-${hash}  (length=${content.length})`);
}
```

Ejecutar tras cualquier cambio a `app/layout.tsx` y re-generar `vercel.json` con el hash actual.

## 🟠 GRAVE #8: Visual regression — verdadero path a la separación

**Estado real:** `ci.yml` tiene un solo job `e2e` que ejecuta `npm run test:e2e`. El script `test:e2e` en `package.json` ejecuta TODOS los specs (visual + funcional + a11y + SEO). Plan A5 dice "extraer del ci.yml" pero el yml solo referencia el script NPM.

**Fix:** La separación es de **scope**, no de yml:

1. Crear tag Playwright `@visual` para specs de regresión visual (e.g. `tests/visual-regression.spec.ts` → describir con `test.describe.serial('Visual', { tag: '@visual' })`).
2. Duplicar script en `package.json`:
   ```json
   "test:e2e": "playwright test --grep-invert @visual",
   "test:visual": "playwright test --grep @visual"
   ```
3. Crear `.github/workflows/visual-regression.yml` con `npx playwright test --grep @visual`.
4. Eliminar `--grep @` del job `e2e` principal (queda por defecto).

## Recomendaciones globales

1. **Cobertura como baseline, no aspiración:** medir primero, decidir threshold después. No escribir el threshold antes de tener el primer reporte.
2. **Tests para todas las nuevas funciones puras:** `calculateReadingTime`, `getAllTags`, `getArticlesByTag`, `extractToc` — todos van a `__tests__/lib/`.
3. **i18n primero, código después:** cualquier string visible al usuario se añade a ambas dictionaries (`es.ts`, `en.ts`) **antes** de los componentes que la usan.
4. **Verificar premises antes de planificar:** si una recomendación se basa en un archivo no leído, leerlo primero. `vercel.json`, `components/mdx.tsx`, `ci.yml`, `package.json` todos verificados aquí.
